# Benchmarks

Cross-library JSON benchmarks for `datavalue-rs`. Compares against
`serde_json`, `simd-json`, `sonic-rs`, and `json-rust` across four workloads
— parse, serialize, access, mutate — over the canonical fixtures used by
the wider Rust JSON ecosystem (`twitter.json`, `citm_catalog.json`,
`canada.json` from the serde-rs json-benchmark suite).

Source: [`benches/compare.rs`](benches/compare.rs). Fixtures:
[`benches/fixtures/`](benches/fixtures/).

## Recommended pairing

`datavalue-rs` ships two value types — pick the one that matches the
workload, and that's what the tables below benchmark:

| Workload | Recommended type | Why |
|---|---|---|
| Parse, serialize, access | **`DataValue`** (arena-bound) | Zero-allocation hot path; `Copy` payloads; no per-node heap traffic. |
| Mutation | **`OwnedDataValue`** (heap-owned) | `DataValue` is immutable by design (the arena is the unit of mutation). `OwnedDataValue` mutates in place via `String` / `Vec`. |

The four result tables surface only the recommended variant per workload —
the parse / serialize / access tables show `datavalue`, the mutate table
shows `OwnedDataValue`. This is the pattern users should follow in their
own code.

```bash

# Short smoke run:
cargo bench --bench compare --features serde_json -- --quick
```

Criterion writes an HTML report under `target/criterion/report/index.html`.

## Hardware

Numbers below were measured on Apple Silicon (`darwin/aarch64`), release
build, single thread, criterion default settings (3 s warm-up, 5 s
measurement). Times shown are medians.

is shallow enough that everyone except sonic_rs ties.

## Mutate — increment retweet_count over twitter.statuses[]

Only `twitter` is exercised here — the patch is a single, well-defined
edit and that's enough to compare mutation costs across DOM shapes.

This is the one workload where `OwnedDataValue` is the recommended
datavalue-rs variant — see the pairing table at the top.

<!-- BENCH:mutate:BEGIN -->
| Library | time | thrpt |
|---|---|---|
| OwnedDataValue (clone + mutate) | 247.02 µs | 2.3809 GiB/s |
| serde_json (clone + mutate) | 325.54 µs | 1.8067 GiB/s |
| simd_json (clone + mutate) | 282.88 µs | 2.0791 GiB/s |
| **json-rust (clone + mutate)** | **121.24 µs** | **4.8510 GiB/s** |
<!-- BENCH:mutate:END -->

All four rows clone the DOM at the start of each iteration, then mutate
the clone in place. `json-rust`'s mutation path is uniquely cheap;
`OwnedDataValue` and `simd_json` clone+mutate in roughly the same ballpark.
`sonic_rs` is omitted — its mutation surface differs enough that fitting a
fair comparable patch in is a follow-up.

## Methodology Notes

- The `datavalue` parse row uses a **reused arena** (`Bump::reset()` between
  iterations) — the workflow this crate is designed for. A separate
  `datavalue (fresh arena)` bench still runs in `benches/compare.rs` for
  completeness; on these fixture sizes the two are within noise (the
  first-chunk allocation is one syscall and disappears against a 500 µs+
  parse). The reused-arena win is meaningful only for sub-microsecond
  parses or large batches.
- All benches use `criterion::black_box` on inputs and outputs.
- `simd_json` requires a mutable byte buffer — each iteration gets a
  fresh `Vec<u8>` via `iter_batched(BatchSize::SmallInput)`, so the bench
  measures parse cost, not buffer reuse.
- The `datavalue` serialize row goes through `DataValue::to_json_string`
  — a native emitter that walks the tree directly, formats numbers with
  `ryu` / `itoa`, and escapes strings via an SWAR scan. The serde-based
  path (via `serde_json::to_string`) still works (the `Serialize` impl
  is intact in `src/ser.rs`) and is the right entry point for non-JSON
  serde sinks like msgpack / flexbuffers.
- `sonic_rs::Value` and `simd_json::OwnedValue` both implement traits
  from the shared `value-trait` crate, which collide on `sonic_rs::Value`
  with `sonic_rs::JsonValueTrait`. The bench imports `simd_json::prelude`
  only inside simd_json bench bodies to avoid the ambiguity.
